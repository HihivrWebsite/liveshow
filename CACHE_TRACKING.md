# 🔍 缓存调试和追踪指南

## 概述

根据日志分析，发现 `room_id=1713546334` 和 `room_id=27628019` 的 `/gift/attention` 接口被频繁调用。

通过本次优化，后端的 `/gift/attention` 端点现在支持**完整的缓存机制**。

## 缓存流程图

```
前端/外部请求
    ↓
GET /gift/attention?room_id=XXXX&month=202604&union=VirtuaReal
    ↓
后端 get_attention_data()
    ↓
    ┌─ 检查缓存键: attention_VirtuaReal_XXXX_202604
    │
    ├─ 缓存命中 (历史数据)
    │   └─→ 返回缓存数据 ✅ [🎯缓存命中]
    │
    ├─ 缓存命中 (当月数据 < 1小时)
    │   └─→ 返回缓存数据 ✅ [🎯缓存命中]
    │
    ├─ 缓存过期 (当月 > 1小时 且 不在1:30-2:00)
    │   └─→ 返回旧缓存 ⏱️ [缓存过期但使用旧数据]
    │
    └─ 缓存未命中或需要刷新
        ↓
        调用外部API: /gift/attention?room_id=XXXX&month=202604
        ↓
        保存到缓存
        ↓
        返回新数据 ✅ [💾缓存保存]
```

## 日志符号说明

| 符号 | 含义 | 说明 |
|------|------|------|
| 📊 | 请求开始 | 接收到客户端请求 |
| 🎯 | 缓存命中 | 直接返回缓存数据，没有外部API调用 |
| 📡 | 缓存未命中 | 首次请求该数据 |
| 🔄 | 缓存过期 | 需要刷新当月数据 |
| 🌐 | API调用 | 向外部服务发起实际请求 |
| 💾 | 缓存保存 | 数据已保存到内存缓存 |
| ⏱️ | 过期但可用 | 缓存已过期但不在刷新时间，使用旧数据 |
| ✅ | 成功 | 操作完成 |
| ❌ | 失败 | 操作失败 |

## 关键缓存策略

### 1. 缓存键格式

```
attention_{union}_{room_id}_{month}

示例：
  attention_VirtuaReal_1713546334_202604
  attention_PSPlive_27628019_202604
```

### 2. 上月数据 (< 202604)

- **过期策略**：永不过期
- **刷新条件**：永不刷新
- **预期日志**：🎯 [缓存命中] 历史数据 - key=xxx (上月数据永久有效)

### 3. 当月数据 (= 202604)

- **过期策略**：1小时
- **刷新条件**：超过1小时且当前时间在 1:30-2:00 之间
- **预期日志**：🎯 [缓存命中] 当月数据 - key=xxx (1小时内有效)

### 4. 缓存失效场景

| 场景 | 处理 |
|------|------|
| 首次请求 | 📡 缓存未命中 → 🌐 API调用 → 💾 保存缓存 |
| 请求相同数据(< 1小时) | 🎯 缓存命中 (直接返回) |
| 请求相同数据(> 1小时) | ⏱️ 缓存过期 (使用旧数据) |
| 请求相同数据(1:30-2:00) | 🔄 缓存过期 → 🌐 API调用 → 💾 更新缓存 |

## 调试方法

### 1. 检查缓存是否命中

运行服务器后，观察日志中是否出现：

```
🎯 [缓存命中] ...
```

这表示缓存正常工作，没有额外的API调用。

### 2. 检查外部API调用

如果频繁看到以下日志：

```
🌐 [API调用] 向外部服务获取数据 - room_id=1713546334, month=202604
```

说明缓存失效太快，需要调整策略。

### 3. 追踪粉丝数计算

搜索以下日志追踪粉丝变化计算的缓存命中情况：

```
📊 [粉丝变化计算] 开始 - room_id=1713546334, ...
✅ [粉丝变化计算] 成功获取attention数据 - room_id=1713546334, month=202604
```

## 性能分析

### 场景1：同一room_id多次请求（相隔 < 1小时）

```
第1次请求: 📡 [缓存未命中] → 🌐 [API调用] → 💾 [缓存保存] (~0.5-2秒)
第2次请求: 🎯 [缓存命中] → ✅ (< 50ms)
第3次请求: 🎯 [缓存命中] → ✅ (< 50ms)
...
第N次请求: 🎯 [缓存命中] → ✅ (< 50ms)
```

**性能提升**: 2-40倍（取决于并发和重复率）

### 场景2：多个room_id（不同主播）

```
room_id=1: 首次请求 → 🌐 API调用 (~1秒)
room_id=2: 首次请求 → 🌐 API调用 (~1秒)
room_id=1: 第2次请求 → 🎯 缓存命中 (< 50ms)
room_id=2: 第2次请求 → 🎯 缓存命中 (< 50ms)
```

## 故障排查

### 问题1：缓存从未命中

**症状**：总是看到 `📡 [缓存未命中]` 或 `🌐 [API调用]`

**可能原因**：
1. 缓存键生成有问题（union参数不同）
2. 缓存大小限制导致旧数据被清除
3. 服务器重启导致内存缓存清空

**解决方案**：
- 检查前端是否传入了 `union` 参数
- 检查 `ATTENTION_CACHE` 的大小限制
- 考虑持久化缓存到数据库

### 问题2：同一room_id重复API调用

**症状**：同一 room_id 多次看到 `🌐 [API调用]`

**可能原因**：
1. 并发请求导致多个线程同时调用API
2. 缓存键生成时 union 参数不一致

**解决方案**：
- 在 `get_attention_cache_data` 中添加并发锁
- 确保所有请求使用相同的 union 参数

## 下一步优化

1. **并发安全**：在缓存更新时添加并发锁，避免多个线程同时调用API
2. **持久化**：考虑将缓存数据保存到Redis或数据库
3. **预加载**：在系统启动时预加载热点数据
4. **监控告警**：当API调用频率异常时发出告警
